Instrucciones del Reto

Analiza cada nivel, identifica la anomalía y redacta el script SQL para corregirlo en tu consola de Neon.tech. Al finalizar cada sección, puedes desplegar la solución para comparar.

Nivel 1: Atomicidad (1FN)

Fácil

Situación: La tabla guarda múltiples cursos en un solo campo de texto.

id_estnombrecursos (Error)
1Ana GarcíaSQL, Python, Java

Reto: Crea el DDL necesario para que cada celda contenga un solo dato.

Justificación:

La 1FN prohíbe grupos repetitivos. Cada registro debe ser único y atómico.

CREATE TABLE Estudiantes_1FN (
    id_est INT,
    nombre TEXT,
    materia TEXT,
    PRIMARY KEY (id_est, materia)
);

Nivel 2: Dependencia Parcial (2FN)

Medio

Situación: El "nombre" depende solo del ID, pero la llave es compuesta (ID + Materia).

Reto: Elimina la redundancia separando los datos del alumno de sus inscripciones.

Justificación:

La 2FN exige que todo atributo no-llave dependa de TODA la llave primaria.

CREATE TABLE Estudiantes (id_est PRIMARY KEY, nombre TEXT);
CREATE TABLE Inscripciones (id_est INT REFERENCES Estudiantes, materia TEXT);

Nivel 3: Transitividad (3FN)

Avanzado

Situación: En la tabla Cursos, la oficina depende del profesor, no directamente del curso.

¿Qué pasa si un profesor cambia de oficina?

Justificación:

La 3FN elimina dependencias entre columnas que no son llaves (A -> B -> C).

CREATE TABLE Profesores (nombre TEXT PRIMARY KEY, oficina TEXT);
ALTER TABLE Cursos DROP COLUMN oficina_profesor;

Nivel 4: Determinantes (BCNF)

Expert

Escenario: Tabla Tutorías (Estudiante, Materia, Tutor).
Regla: Un tutor solo enseña una materia.

Problema: El Tutor determina la Materia, pero no es llave.

Justificación:

En BCNF, cada determinante debe ser una llave candidata.

CREATE TABLE Asignacion_Tutor (tutor_id INT PRIMARY KEY, materia TEXT);
CREATE TABLE Sesiones_Tutor (est_id INT, tutor_id INT REFERENCES Asignacion_Tutor, PRIMARY KEY (est_id, tutor_id));

Nivel 5: Multivaluados (4FN)

Pro

Escenario: Un profesor tiene varios cursos y varios títulos académicos independientes.

Evita la explosión combinatoria de registros redundantes.

Justificación:

La 4FN separa hechos independientes representados en relaciones de muchos a muchos.

CREATE TABLE Prof_Cursos (prof_id INT, curso_id INT);
CREATE TABLE Prof_Titulos (prof_id INT, titulo TEXT);

Nivel 6: Join-Project (5FN)

Legendary

Escenario: Relación ternaria cíclica entre Proveedor, Producto y Proyecto.

Justificación:

La 5FN evita anomalías de unión descomponiendo la tabla en todas sus proyecciones posibles.

CREATE TABLE Prov_Prod (prov_id INT, prod_id INT);
CREATE TABLE Prod_Proj (prod_id INT, proj_id INT);
CREATE TABLE Prov_Proj (prov_id INT, proj_id INT);
